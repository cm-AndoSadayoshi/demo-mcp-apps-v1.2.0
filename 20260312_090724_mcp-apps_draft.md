# MCP Apps v1.2.0の新API「readServerResource / listServerResources」を使ってリソースブラウザUIを作ってみた

## はじめに

MCP Apps は、Model Context Protocol（MCP）初の公式拡張機能です。AIチャット内にダッシュボード、フォーム、グラフなどのインタラクティブなUIを埋め込むための標準仕様として、2026年1月にAnthropicとOpenAIの共同で策定されました。

2026年3月6日にリリースされた **v1.2.0** では、View（UI側）からサーバーのMCPリソースに直接アクセスできる新APIが追加されました。従来はツール呼び出し経由でしかデータを取得できませんでしたが、`listServerResources()` と `readServerResource()` を使うことで、よりリッチなUI体験を構築できるようになっています。

結論から言うと、v1.2.0の新APIにより、ViewからMCPリソースに直接アクセスできるようになり、ツール呼び出しに依存しないリッチなUI構築が格段に容易になりました。

この記事では、これらの新APIを使って「リソースブラウザ」MCP Appを実際に作ってみた手順と結果をお伝えします。MCPサーバーを触ったことがある方、MCP Appsに興味がある方が対象です。

## MCP Apps のおさらい

MCP Apps のアーキテクチャは、以下の3つのコンポーネントで構成されています。

| コンポーネント | 役割 | 具体例 |
|---|---|---|
| **Server** | 標準MCPサーバー。ツールとUIリソースを宣言し、HTMLでUIを定義 | Node.js上で動作するMCPサーバー |
| **Host** | チャットクライアント。サーバーに接続し、Viewをiframeで埋め込み、通信をプロキシ | Claude Desktop, ChatGPT, VS Code |
| **View** | サンドボックスiframe内で動作するUI。HostからデータをJSON-RPCで受信 | ブラウザ上のHTML/JS |

[TODO: スクリーンショット - MCP Apps アーキテクチャ図（Server → Host → View の関係を示す図）]

ポイントは `ui://` URIスキームです。サーバーはUIテンプレート（HTML/JS）を `ui://` スキームで宣言し、ツール定義の `_meta.ui.resourceUri` フィールドでそのUIを関連付けます。ホストはリソースをフェッチしてサンドボックスiframeで表示し、JSON-RPC over postMessageで双方向通信を実現します。

MCP Appsはオプショナルな拡張のため、既存のMCP実装に影響はありません。UI非対応のホスト向けには、テキストフォールバックを提供することが推奨されています。

現在の対応クライアントは、Claude（Web/Desktop）、ChatGPT、VS Code GitHub Copilot、Goose、Postman、MCPJamなどです。

## v1.2.0 で何が変わったのか

v1.2.0（2026年3月6日リリース）の目玉は、ViewからサーバーのMCPリソースへ直接アクセスできるAPIの追加です。

### `listServerResources()` — リソース一覧の取得

ViewからサーバーがMCPで公開しているリソースの一覧を取得できます。従来、View側でリソースの存在を知るには、ツール呼び出し経由でサーバーに問い合わせる必要がありました。この新APIにより、Viewが自律的にリソースを発見できるようになりました。

### `readServerResource()` — リソース内容の読み取り

指定したリソースURIの内容をViewから直接読み取ります。テキスト、JSON、バイナリなど、サーバーが公開するあらゆるリソースにアクセスできます。

### `ui/download-file` — ホスト経由ファイルダウンロード

UIからファイルをダウンロードする機能です。ホストが仲介することで、サンドボックス内のViewからでも安全にファイルをローカルに保存できます。

これらの追加により、MCP AppsのViewは「ツールの結果を表示するだけ」から「サーバーリソースを自由に探索・閲覧できるインターフェース」へと進化しました。

## 開発環境のセットアップ

以下の環境で開発を行いました。

- Node.js: v21.2以上（`import.meta.dirname` を使用するため）
- TypeScript: 5.x
- Vite: 6.x
- `@modelcontextprotocol/ext-apps`: v1.2.2（最新安定版）
- `@modelcontextprotocol/sdk`: 最新版

まず、プロジェクトを作成します。

```bash
mkdir mcp-resource-browser && cd mcp-resource-browser
npm init -y

# ESモジュールを有効化
npm pkg set type=module

npm install @modelcontextprotocol/sdk @modelcontextprotocol/ext-apps
npm install -D typescript vite @types/node
```

プロジェクト構成は以下のようにしました。

```
mcp-resource-browser/
├── package.json
├── tsconfig.json
├── src/
│   └── server.ts          # MCPサーバー（リソース登録 + ツール定義）
├── ui/
│   ├── index.html          # View（リソースブラウザUI）
│   └── app.ts              # View側のロジック
└── vite.config.ts          # Vite設定（UIバンドル用）
```

`tsconfig.json` は以下のように設定しました。

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",

    "rootDir": "./src",

```

## サーバー側の実装

MCP Appsでは、通常のMCPサーバーにUIリソースとアプリ対応ツールを追加する形で実装します。v1.2.0以降のSDKでは `registerAppTool()` と `registerAppResource()` というヘルパー関数が提供されており、ボイラープレートを大幅に削減できます。

```typescript
// src/server.ts
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import {
  registerAppTool,
  registerAppResource,
} from "@modelcontextprotocol/ext-apps";
import * as fs from "fs/promises";
import * as path from "path";

const server = new McpServer({
  name: "resource-browser",
  version: "1.0.0",
});

// サンプルデータとして公開するリソースを登録
const sampleResources = [
  {
    uri: "data://docs/getting-started",
    name: "Getting Started Guide",
    description: "MCP Appsの基本的な使い方",
    mimeType: "text/markdown",
    content: "# Getting Started\n\nMCP Appsを始めるには...",
  },
  {
    uri: "data://docs/api-reference",
    name: "API Reference",
    description: "ext-apps SDKのAPIリファレンス",
    mimeType: "text/markdown",
    content: "# API Reference\n\n## connect()\n\nHostに接続します...",
  },
  {
    uri: "data://config/settings",
    name: "Server Settings",
    description: "サーバー設定（JSON）",
    mimeType: "application/json",
    content: JSON.stringify(
      { theme: "dark", maxConnections: 10, debug: false },
      null,
      2
    ),
  },
];

// MCPリソースとして登録
for (const res of sampleResources) {
  server.resource(res.uri, res.name, async () => ({
    contents: [
      {
        uri: res.uri,
        mimeType: res.mimeType,
        text: res.content,
      },
    ],
  }));
}

// UIリソース（HTML）を ui:// スキームで登録
const uiHtml = await fs.readFile(
  path.join(import.meta.dirname, "../ui/dist/index.html"),
  "utf-8"
);

registerAppResource(server, {
  uri: "ui://resource-browser/index",
  name: "Resource Browser UI",
  content: uiHtml,
});

// UI対応ツールを登録
registerAppTool(server, {
  name: "browse-resources",
  description:
    "サーバーのリソースをインタラクティブに閲覧します",
  inputSchema: {},
  uiResourceUri: "ui://resource-browser/index",
  handler: async () => {
    // テキストフォールバック（UI非対応ホスト用）
    const resourceList = sampleResources
      .map((r) => `- ${r.name}: ${r.description}`)
      .join("\n");
    return {
      content: [
        {
          type: "text",
          text: `利用可能なリソース:\n${resourceList}`,
        },
      ],
    };
  },
});

// サーバー起動
const transport = new StdioServerTransport();
await server.connect(transport);
console.error("Resource Browser MCP Server started");
```

ここでのポイントは3つあります。

1つ目は **`registerAppResource()`** です。`ui://` スキームのリソースを登録するヘルパーで、HTMLコンテンツをUIリソースとしてサーバーに登録します。

2つ目は **`registerAppTool()`** です。ツール定義と `_meta.ui.resourceUri` の関連付けを一括で行えます。`uiResourceUri` を指定するだけでOKです。

3つ目は **テキストフォールバック** です。`handler` の戻り値にテキストコンテンツを含めることで、MCP Apps非対応のホストでも結果が表示されます。これは仕様で推奨されているベストプラクティスです。

次に `ui/app.ts` を実装します。ここが今回の核心部分です。

```typescript
// ui/app.ts
import { connect } from "@modelcontextprotocol/ext-apps/view";

async function main() {
  // HostへPostMessageTransport経由で接続（デフォルト）
  const app = await connect();

  const statusEl = document.getElementById("status")!;
  const listEl = document.getElementById("resourceList")!;
  const previewEl = document.getElementById("preview")!;

  // ① listServerResources() でリソース一覧を取得
  statusEl.textContent = "リソース一覧を取得中...";
  const resources = await app.listServerResources();
  statusEl.textContent = `${resources.length}件のリソースが見つかりました`;

  // リソース一覧をリスト表示
  for (const resource of resources) {
    const item = document.createElement("div");
    item.className = "resource-item";
    item.innerHTML = `
      <div class="name">${resource.name ?? "Unnamed"}</div>
      <div class="uri">${resource.uri}</div>
    `;

    item.addEventListener("click", async () => {
      // アクティブ状態の更新
      document.querySelectorAll(".resource-item").forEach((el) => {
        el.classList.remove("active");
      });
      item.classList.add("active");

      // ② readServerResource() でリソース内容を読み取り
      statusEl.textContent = `${resource.uri} を読み込み中...`;
      try {
        const result = await app.readServerResource(resource.uri);
        const content = result.contents[0];
        const text =
          "text" in content ? content.text : "(バイナリコンテンツ)";

        previewEl.innerHTML = `
          <div style="margin-bottom: 12px;">
            <strong>${resource.name}</strong>
            <span style="margin-left: 8px; font-size: 12px; color: #888;">
              ${content.mimeType ?? "unknown"}
            </span>
            <button class="download-btn" id="downloadBtn"
                    style="float: right;">
              ダウンロード
            </button>
          </div>
          <pre>${escapeHtml(text)}</pre>
        `;

        // ③ ui/download-file でファイルダウンロード
        const downloadBtn =
          document.getElementById("downloadBtn")!;
        downloadBtn.addEventListener("click", async () => {
          const ext = content.mimeType === "application/json"
            ? ".json"
            : ".md";
          const filename =
            (resource.name ?? "resource").replace(/\s+/g, "-") +
            ext;

          await app.requestFileDownload({
            filename,
            content: text,
            mimeType: content.mimeType ?? "text/plain",
          });
          statusEl.textContent = `${filename} のダウンロードをリクエストしました`;
        });

        statusEl.textContent = `${resource.uri} を表示中`;
      } catch (err) {
        previewEl.innerHTML = `<p style="color: red;">
          読み込みエラー: ${(err as Error).message}
        </p>`;
        statusEl.textContent = "エラーが発生しました";
      }
    });

    listEl.appendChild(item);
  }
}

function escapeHtml(text: string): string {
  return text
    .replace(/&/g, "&amp;")
    .replace(/</g, "&lt;")
    .replace(/>/g, "&gt;");
}

main().catch(console.error);
```

コードの流れを整理すると、以下の3ステップです。

1. **`connect()`** でHostに接続。v1.2.0以降、`PostMessageTransport(window.parent)` がデフォルトなので引数は不要です
2. **`listServerResources()`** でサーバーが公開しているMCPリソースの一覧を取得し、左ペインにリスト表示
3. リソースをクリックすると **`readServerResource(uri)`** で内容を取得し、右ペインにプレビュー表示。ダウンロードボタンを押すと **`requestFileDownload()`** でホスト経由のファイルダウンロードをリクエスト

`connect()` が返すオブジェクトに `listServerResources()` や `readServerResource()` が生えているので、非常に直感的に使えました。なお、`requestFileDownload()` は内部的に `ui/download-file` JSON-RPCメソッドを呼び出すSDKのヘルパーメソッドです。公式ドキュメントで `ui/download-file` と記載されている箇所は、SDK経由ではこのメソッド名に対応します。

## ハマりポイント・注意事項

実際に開発する中でいくつかハマったポイントがありました。今後MCP Appsを触る方は参考にしてください。

### CSP設定の罠 — デフォルトは「外部接続不可」

MCP Appsのセキュリティモデルでは、サーバーがUIに必要な外部ドメインをメタデータで明示的に宣言しなければ、**外部への接続は一切ブロック**されます（restrictive by default）。

例えば、UIからCDN経由で外部ライブラリを読み込もうとすると、何のエラーメッセージもなく失敗します。CDNを使いたい場合は、サーバー側で `csp` メタデータにドメインを宣言する必要があります。

```typescript
registerAppResource(server, {
  uri: "ui://my-app/index",
  name: "My App",
  content: htmlContent,
  metadata: {
    csp: {
      connectSrc: ["https://api.example.com"],
      scriptSrc: ["https://cdn.jsdelivr.net"],
    },
  },
});
```

個人的には、Viteでバンドルしてすべてインラインに含めてしまうのが最もシンプルでハマりにくいアプローチだと感じました。

### zod バージョン互換性 — v1.2.2以降を使うこと

v1.2.0とv1.2.1では、スキーマインポート時にSDKの zod APIバージョンとの互換性問題がありました。具体的には、ビルド時に以下のようなエラーが出ることがあります。

```
Error: Cannot find module '@modelcontextprotocol/sdk/dist/zod.js'
```

これはv1.2.2で修正されています。`@modelcontextprotocol/ext-apps` は **必ずv1.2.2以降** を使ってください。

```bash
npm install @modelcontextprotocol/ext-apps@latest
```

### 自動リサイズの注意点

MCP AppsにはViewのコンテンツに合わせてiframeの高さを自動調整する機能があります。v1.2.0でビューポート高さの計算バグが修正されましたが、動的にコンテンツが増えるケース（例: 非同期でデータを読み込んで表示する場合）では、自動リサイズが追いつかないことがあります。

その場合は、コンテンツ更新後に明示的にリサイズを通知する実装を検討してください。

### sandbox制約を意識する

ViewはサンドボックスiframeとMCP Apps仕様で動作するため、以下の制約があります。

| 制約 | 詳細 |
|---|---|
| `allow-scripts` + `allow-same-origin` の同時使用不可 | セキュリティ上禁止。これによりクッキーやlocalStorageにアクセスできない |
| Clipboard API | サンドボックス内では動作しない場合がある |
| ポップアップ・新規タブ | `window.open()` は基本的にブロックされる |
| Top-level navigation | `allow-top-navigation-by-user-activation` が必要な場合がある |

シンプルなUIであれば問題になりませんが、複雑な機能を実装しようとするとこれらの制約に引っかかることがあります。

## まとめ

MCP Apps v1.2.0で追加された `listServerResources()` と `readServerResource()` を使って、サーバーのMCPリソースをインタラクティブに閲覧できる「リソースブラウザ」を作ってみました。

今回のポイントをまとめると以下の通りです。

- **`listServerResources()`** でViewからリソース一覧を自律的に取得できるようになった
- **`readServerResource()`** でツール呼び出しを介さずにリソース内容を直接読み取れる
- **`requestFileDownload()`** でホスト経由の安全なファイルダウンロードが可能
- `registerAppTool()` / `registerAppResource()` ヘルパーでサーバー側の実装が簡潔に
- CSP設定やzodバージョン互換性など、開発時に気をつけるポイントもいくつかある

これらの新APIにより、MCP Appsは「ツールの結果を表示するUI」から「サーバーリソースを自由に探索できるインタラクティブなアプリケーション」へと進化しつつあります。今後、さらに多くのクライアントがMCP Appsに対応していくことで、AIチャット内でのUI体験がますますリッチになっていくことが期待されます。
